AI On-Call Triage Agent

A triage agent for production incidents that pairs deterministic severity rules with LLM reasoning — and is architected so the model can never quietly downgrade a severity the rules have already flagged.

Stack
FastAPI · Python · Claude (Anthropic API)
Role
Product design & architecture
Status
Working prototype

The problem

On-call rotations absorb a steady stream of alerts, and a large share of them follow recognizable patterns — a familiar severity, a familiar category, a familiar set of next steps. Based on real alert pattern data from an enterprise on-call environment, an estimated 60–70% of alerts were preventable or followed a pattern that could have been triaged automatically before reaching a human.

The product question wasn't "can an LLM triage an incident" — it clearly can. The harder question was: how do you let an LLM help with triage without letting it become the thing that decides whether a SEV1 outage gets treated as a SEV1?

The approach: rules constrain the model, not the other way around

The triage service runs a deterministic pre-triage pass first — keyword and pattern-based detection of severity and category from the incident title, description, logs, and recent changes. This pass runs before the LLM is called, and its output is treated as ground truth for the things it's confident about.

The LLM (Claude) is then given the incident plus the deterministic pre-triage result, and asked to produce a structured assessment — executive summary, immediate actions, recommended owner, escalation targets — as strict JSON. After the model responds, the service re-applies the deterministic rules as a safety check: if the rules detected SEV1, the final result is forced to SEV1 regardless of what the model returned, and any escalation targets the rules require are merged in even if the model omitted them.

# Severity override — rules win, the model can't downgrade a SEV1 if p["severity"] == "SEV1" and parsed.get("severity") not in ["SEV1", "UNKNOWN"]: parsed["severity"] = "SEV1" # Escalation targets are merged, never replaced if p["escalation_needed"]: parsed["escalation_needed"] = True parsed["escalation_targets"] = sorted(set( parsed.get("escalation_targets", []) + p["escalation_targets"] ))

The API also exposes a /triage/mock endpoint that returns deterministic-only results with no model call — useful for testing the rules layer in isolation, and for environments where calling out to an LLM isn't appropriate for a given incident category.

The experience: an AI teammate that asks before it acts

To make this tangible beyond an API contract, the workflow is also modeled as an incident-response assistant operating inside a team chat — investigating an alert, posting a structured diagnosis, surfacing a similar past incident, and recommending a mitigation. Critically, the recommended action is not executed automatically — it sits behind an explicit human approval step before anything changes in production.

Why the architecture matters

This is the part of the project that's most relevant to a product role focused on AI agent governance: the same pattern — deterministic guardrails wrapping a probabilistic model, with a human approval gate before any state-changing action — is exactly the control structure that AI agents need when they're given access to production systems, IAM platforms, or customer data. An on-call triage agent is a low-stakes place to prove this pattern works before applying it to identity and access decisions.
